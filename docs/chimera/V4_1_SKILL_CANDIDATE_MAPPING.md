## 0. Reading guide

This document is a **mapping plan**, not a roster of finished work. It pairs a list of in-flight candidates (work-lite, language, exam, media-lite, agent-security, …) with the Chimera v4.1 invariants and proposes, for each candidate:

1. **Parent `carrier_pack`(s)** — which existing P0/P1 pack(s) from [RFC-009 §3 / §4](../rfcs/RFC-009-chimera-v4.1.md) this candidate sits under or composes with.
2. **Namespace + proposed canonical filename** — what the candidate would be called if (and only if) it ever passes the ten validation criteria of [RFC-009 §8](../rfcs/RFC-009-chimera-v4.1.md).
3. **Target carrier (user)** — who would actually carry this pack (a learner, a developer, a maintainer, a guardian, …). No PII (RFC-009 §8.4).
4. **Competencies to map** — which authoritative-framework anchors the candidate must resolve to (ESCO / WEF / O\*NET / DigComp / EQF / CEFR / NICE / ENISA / CIS / SFIA — see [`docs/rfcs/chimera/frameworks/README.md`](../rfcs/chimera/frameworks/README.md) §1).
5. **Gates + human veto** — RFC-002 defaults, `human_authority` posture (always `final_decision_owner ∈ {human_carrier, human_carrier_with_guardian}`).
6. **Source / evidence policy** — which RFC-002 §8b grounding shape and RFC-003 `verification_artifacts[]` reference the candidate would use.
7. **Size tier** — Lot A (lightweight, near current Chimera size; `tokens_estimate ≤ ~900`) or Lot B (advanced; +50% size allowance, `tokens_estimate ≤ ~1,350`, requires compact index / on-demand loading).
8. **Readiness status** — `candidate_mapped` (framework anchors identified, scaffold not yet authored), `needs_mapping` (framework anchors insufficient or contested — explicitly NOT `candidate_mapped`), or `ship_ready` (validation §8 fully satisfied — none of the candidates below is in this state).

The terms `carrier_pack`, `competency_pack`, `host_skill`, `base_transversal_core`, and `human_authority_layer` are used per the canonical Chimera vocabulary in [RFC-009 §0.1](../rfcs/RFC-009-chimera-v4.1.md). The forbidden phrasings of [RFC-009 §0.1.2](../rfcs/RFC-009-chimera-v4.1.md) apply throughout this document.

## 0.1 What this document does NOT do

- It does **not** create any `.klickd` pack file. All references to `x.klickd/<name>` below are **planning identifiers**, not artefacts on disk. No new file appears under `examples/v4/starter-skills/`, no new file appears under `docs/rfcs/chimera/packs/fixtures/`, no entry is added to `examples/v4/starter-skills/manifest.json`.
- It does **not** publish anything to npm / PyPI / Zenodo / IANA.
- It does **not** tag a release or update `CHANGELOG.md`.
- It does **not** introduce a `/klickdskill` listing, search index, API endpoint, or download surface for any of the candidates below. The no-catalog rule of [RFC-009 §7](../rfcs/RFC-009-chimera-v4.1.md) applies in full.
- It does **not** include any Klickd.app-specific carrier (`klickdapp.lu.klickd`, `klickdapp.fr.klickd`, `klickdapp.be.klickd`, `klickdapp.de.klickd`, "Kai Tutor", "Kai Mentor", any other `kai.*` host-side skill). Those are Klickd.app product artefacts that live under [`examples/v4/klickdapp-skills/`](../../examples/v4/klickdapp-skills/) and are explicitly out of scope here. The public `/klickdskill` catalog and any planning document under `docs/chimera/` MUST NOT reference them as candidate Chimera packs.
- It does **not** alter the v4.0 envelope, the existing canonical P0 starters (`user.klickd`, `student.klickd`, `research.klickd`, `coding.klickd` under [`examples/v4/starter-skills/`](../../examples/v4/starter-skills/)), or the P1 candidate identifiers fixed in [RFC-009 §4](../rfcs/RFC-009-chimera-v4.1.md) (`x.klickd/work`, `x.klickd/creator`, `x.klickd/gaming`, `x.klickd/bridge`, `x.klickd/mission`).

## 0.2 Sharp boundary: `candidate_mapped` vs `ship_ready`

| Status | Meaning | What is required | What is NOT yet required |
|---|---|---|---|
| `needs_mapping` | The candidate has not yet been anchored to a sufficient set of authoritative frameworks; reviewers cannot evaluate it against [RFC-009 §8.1](../rfcs/RFC-009-chimera-v4.1.md). | A `Notes / open questions` line in §1 / §2 explaining what is missing. | Anything else. The candidate stays parked until anchors land. |
| `candidate_mapped` | The candidate has an identified parent pack, a proposed canonical filename, a target carrier, a candidate set of framework anchors, a gate posture, and a size-tier slot. | All seven fields populated; reviewer can argue with scope, name, and anchors. | A scaffold under `docs/rfcs/chimera/packs/`. An offline SKOS/JSON-LD bundle. A strict JSON Schema. A round-trip vector. A `router_cost` row computed against real bytes. |
| `ship_ready` | The candidate has passed all **ten** validation criteria of [RFC-009 §8](../rfcs/RFC-009-chimera-v4.1.md). Eligible for a future promotion RFC + checklist gate (but **not** automatically eligible for catalog exposure — that is a separate decision per RFC-009 §7). | Everything in `candidate_mapped` **plus** scaffold, framework anchors fully resolved, offline bundle bytes, strict JSON Schema, round-trip vector, deterministic `router_cost`, static `forbidden_fields` check, pair-reviewer sign-off. | — |

**No candidate in this document is `ship_ready`.** A few — notably `personal-finance` / `budget` overlap, `family`, `wellbeing-lite` — are explicitly `needs_mapping`. The rest are `candidate_mapped`. Promotion past `candidate_mapped` is a separate per-pack RFC + scaffold under [`docs/rfcs/chimera/packs/`](../rfcs/chimera/packs/), gated by [`docs/rfcs/ACCEPTANCE_CHECKLIST_V4.md`](../rfcs/ACCEPTANCE_CHECKLIST_V4.md).

| Tier | Audience | `tokens_estimate` ceiling | Loading strategy | Examples (P0/P1, already mapped) |
|---|---|---|---|---|

The size tier is **descriptive**, not normative — it is a planning hint so reviewers can spot a candidate that would blow the seven-pack ceiling ([RFC-009 §5.3 / §5.4](../rfcs/RFC-009-chimera-v4.1.md)) if loaded alongside the always-on `x.klickd/user`.

## 0.4 Naming convention used in this document

Per Vince's review, ambiguous names like `work-lite`, `student-lite`, `legal-lite` are **planning-only nicknames**; they MUST NOT appear as canonical `x.klickd/<name>` identifiers. Each candidate row below carries:

- a **working nickname** (the term the user requested in the planning conversation — `work-lite`, `language`, `exam`, …), and
- a **proposed canonical filename** under the strict `x.klickd/<name>` namespace, picked to:
  1. preserve already-mapped canonical files where they exist (`x.klickd/user`, `x.klickd/student`, `x.klickd/research`, `x.klickd/coding`, `x.klickd/security`, `x.klickd/legal`, the P1 set fixed in RFC-009 §4),
  2. avoid clashing with `core.<agent>.klickd` host-side cores from [RFC-006](../rfcs/RFC-006-agent-core.md) (any `core.*` name is host-side, never a carrier_pack),
  3. avoid clashing with Klickd.app product names (no `klickdapp.*`, no `kai.*`).

Where Vince's working name maps cleanly onto a parent pack, the canonical filename is the parent's identifier and the working name becomes a *sub-area* of that pack (e.g. `language` → `x.klickd/student` `language_proficiency` block; `exam` → `x.klickd/student` `exam_targets[]` block). Where Vince's working name names a new sibling pack, the canonical filename is a new `x.klickd/<name>` identifier and is flagged as a **proposed P1+ addition** that must clear an RFC step before authoring.

### 1.1 Lot A notes & open questions

- **Sub-profiles vs new packs.** `work-lite` (A1) and `media-lite` (A4) are explicitly **sub-profiles of the existing P1 packs** `x.klickd/work` and `x.klickd/creator`. They are NOT new sibling packs. Future authoring lands inside the parent pack's scaffold under `docs/rfcs/chimera/packs/work.md` / `creator.md` with a `profile: "lite"` selector (illustrative; no schema yet).
- **`language` (A2) and `exam` (A3) are NOT separate packs.** They are sub-areas of `x.klickd/student` (already pinned in [`docs/rfcs/chimera/packs/student.md`](../rfcs/chimera/packs/student.md) §2). The mapping here is so reviewers do not propose them as siblings later.
- **`personal-finance` / `budget` (A5/A6) blocked on framework anchor.** No EU SKOS-published financial-literacy framework exists with DigComp maturity. The current draft proposes citing OECD/INFE 2020 toolkit and EBA financial-literacy joint communications as **citations**, not as `frameworks[]` schemes. Until reviewers agree a path, both stay `needs_mapping`.
- **`wellbeing-lite` (A14) blocked on framework anchor.** Same shape of problem as A5 for the health-literacy side. The candidate cannot ship before the clinical-vs-non-clinical line is anchored (the `block` gate above is the minimum; the framework anchor is the open question).
- **`family` (A15) blocked on framework + PII-of-others.** A `family` pack that names other people instantly leaks PII under RFC-009 §8.4 unless an explicit per-person consent shape is part of the pack. The current draft requires this shape; until one is reviewed, the candidate stays `needs_mapping`.
- **`crypto-lite` (A8) keeps strict gates from `x.klickd/security`.** The pack inherits `security`'s blast-radius reasoning; reviewers MUST NOT relax the `block` gate on seed-phrase / private-key surfaces.

### 2.1 Lot B notes & open questions

- **Compact-index loading is the size-tier discipline.** Lot B candidates that exceed Lot A's ~900-token ceiling MUST adopt the compact-index strategy (manifest top-level keys + framework IDs + competency IDs + gates + `human_authority` + `router_cost` in prompt; full bodies lazy). Any candidate whose compact index alone exceeds ~1,350 tokens MUST split into two packs rather than break the ceiling.
- **All Lot B candidates are `candidate_mapped`.** None is `ship_ready`. Each one needs its own per-pack RFC + scaffold under `docs/rfcs/chimera/packs/<name>.md` before any authoring. The order of authoring is **not** the order of this table; it is decided by the P1+ RFC sequencing in [`README_V4_1.md`](./README_V4_1.md) §3.
- **`agent-security` (B1), `iam-endpoint` (B3), `release-engineer` (B4) compose with `x.klickd/security` (P0).** They MUST NOT redefine or lower any default from `security`; they MAY raise gates only.
- **`eu-ai-act` (B6), `gdpr-readiness` (B7), `contract-review` (B8), `privacy-product` (B9), `rights-guard` (B18) compose with `x.klickd/legal` (P0).** Same rule — they MAY raise gates, never lower. Every one of them carries a mandatory escalation to qualified counsel for definitive decisions.
- **`evidence-desk` (B10), `policy-analyst` (B11), `second-brain` (B12), `literature-review` (B13) compose with `x.klickd/research` (P0).** The carrier-vs-skill rule of [RFC-009 §5.1.1](../rfcs/RFC-009-chimera-v4.1.md) applies — these packs carry the carrier's **state** (what they have done, what gates apply, what consent posture they hold); the matching **method** (how to investigate, how to write a brief, how to write atomic notes, how to run a PRISMA review) lives on the host side as a `host_skill` (`skill.research.assistant` for the generic case, with possible host-specific specialisations the host vendor chooses).
- **`drone` (B15), `mission-control` (B16)** introduce **operations-time-critical gates**. The `block` defaults above are minimums; reviewers MUST audit them before the candidates move past `candidate_mapped`.
- **`game-design` (B17), `rights-guard` (B18)** sit at the intersection of creator / legal / coding. They are listed separately so authors do not silently fold them into `x.klickd/creator` (P1) without an RFC step.
