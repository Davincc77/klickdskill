> **This RFC is non-normative.** It describes a *future* track on top of `.klickd v4`. It does **not** bind any current SDK, schema, reader or writer, and is **NOT** in scope for the `v4.0.0` GA P0 backlog defined in [`docs/roadmap/ROAD-TO-V4-GA.md`](../roadmap/ROAD-TO-V4-GA.md). v3.x and v4-preview readers MUST IGNORE every field listed below if they encounter it. v4 readers that do not understand `usage_profile` MUST preserve it verbatim on round-trip (SPEC §33.7).

---

## 1. Motivation

`.klickd` files today carry a generic, full-surface profile: memory, growth, personality, ethics, agent_instructions, and (in v4) media_profile, verification_gates, etc. This works well for a returning user with a curated profile, but creates two real frictions:

1. **First-run bloat.** A brand-new `.klickd` file generated by a tool that does not know *why* the user wants one ends up either (a) empty and unhelpful, or (b) over-populated with sections the user does not need (e.g. `media_profile` for a legal researcher; `claim_sources.prefer` for a casual learner). A user is asked to consent, to fill, or to migrate fields they do not care about. This is the opposite of "never break the soul" — it adds soul the user never asked for.
2. **In-session ambiguity.** Inside a single session, a user's intent shifts (a developer asks a legal question; a creator asks a finance question). The agent has no portable, format-level signal to choose which skill/profile facet best serves the current turn, beyond reading the entire `.klickd` and guessing. The guess is opaque, unauditable, and not consistent across providers.

This RFC proposes a single, additive payload section — `usage_profile` — that solves both:

- **At first run**, a generator MAY ask the user to pick a *purpose* (learner, creator, developer, legal, finance, health, research, agent_core, …). The choice writes a minimal, focused base into `usage_profile.active_profile` and an empty / opt-in scaffold for the other facets. **Progressive disclosure**: the file stays small until the user adds material.
- **In session**, a `.klickd`-aware host MAY consult `usage_profile.routing_rules` and `usage_profile.capabilities` to choose which skill/facet is best suited to the current user turn, and log the choice in `profile_switch_log` with evidence/confidence — without ever storing the raw user prompt in the core file.

This is **complementary** to:

- `profile_kind` (SPEC §33.3): a single coarse discriminator for the file's *role* (`learner` / `agent` / `team` / `robot`). `usage_profile` is finer-grained, *within* a `profile_kind` (a learner can have purpose `legal-research` today and `creative-writing` tomorrow).
- `onboarding_trigger` (SPEC §29b): a string enum telling an agent how to handle a missing profile. `usage_profile` is what the agent populates *as a result of* an onboarding flow.
- RFC-006 (`agent_core`): describes the agent's *own* operating context. `usage_profile` lives in the **user** profile and tells *that* user's agents which facet to engage. The two are orthogonal and follow the same `one-file-one-role` invariant.

## 2. Scope

**In scope (v1 of this RFC):**

- A payload-level `usage_profile` object describing the user's selected purpose(s), capabilities they care about, preferred domains, tool permissions, routing rules, the currently active profile, an append-only switch log, and a per-entry confidence/evidence model.
- A **progressive-disclosure principle**: the base `.klickd` ships with the minimum required for the chosen purpose; other facets are explicitly opt-in.
- A first-run **purpose selection** UX contract (illustrative; the prompt template is host-controlled).
- An in-session **skill routing** contract: how a host SHOULD pick a facet for the current turn and how it MUST log the pick.
- A **privacy invariant**: `usage_profile` MUST NOT store raw user prompts, transcripts, or any user-typed text beyond the user's own self-declared metadata (purpose label, capability tags). Routing evidence is structured, not free-form.
- A `profile_kind`-compatibility table: which `usage_profile.active_profile` values are admissible under which `profile_kind`.
- Relation rules with RFC-002 (`verification_gates`): a `usage_profile` MAY suggest stricter defaults for a sensitive purpose (e.g. `health`, `legal`, `finance`), but the user's `verification_gates` remain authoritative (user wins on conflict, same invariant as RFC-006 §4 #5).

**Out of scope (v1, deferred):**

- A registry / marketplace of purposes. v1 ships a closed enum of starter purposes; extensibility is via `x_purpose_*` per the existing `.klickd` unknown-field rule.
- Cross-user collaborative routing (team-level skill suggestions). Out of v1 scope; one user, one `.klickd`.
- Probabilistic / ML-based routing. v1 documents a *deterministic* routing-rules shape; ML scoring is host-internal and only its *output* (confidence + evidence references) is recorded.
- Strict JSON Schema for `usage_profile`. v1 of this RFC describes the shape; strict schema arrives only if and when the RFC is promoted past `Proposed`.
- Migration of v3.x files to add a `usage_profile` automatically. Per RFC-004 ("never break the soul"), migration is **non-destructive and opt-in**: a migrator MAY *propose* a default `usage_profile.active_profile = "learner"` (matching the v3.x implicit default) but MUST NOT write it without user confirmation.

## 3. Forward compatibility

This RFC targets a `.klickd v4+` payload extension. It does **not** modify v3.x, and it does **not** modify the normative surface of `v4.0.0` GA. Specifically:

- A `v4.0.0` GA writer MAY emit `usage_profile` only if the RFC is at least `Proposed`; otherwise it stays strictly on the post-GA track.
- A `v4.0.0` GA reader that does not understand `usage_profile` MUST preserve it verbatim on round-trip (SPEC §33.7, RFC-004).
- A v3.x reader presented with a file carrying `usage_profile` MUST behave as it does for any unknown v4 field (preserve verbatim if it round-trips; otherwise surface `KLICKD_E_VERSION` rather than silently dropping).
- A v4 reader that *does* understand `usage_profile` MUST NOT use it as a *floor* for the user's `verification_gates` (gates win; see §6).

## 4. Decisions already resolved

The following decisions form the baseline. Reviewers should attack the **right** target.

1. **Progressive disclosure is the default.** A fresh `.klickd` written for a single purpose contains *only* the sections that purpose needs. Adding a second purpose adds its sections; it does not preallocate every possible field. The `.klickd` stays small until the user grows it.
2. **Purpose is user-owned.** `usage_profile` lives in the user's `.klickd`, not in `agent_core` (RFC-006). An agent core MAY suggest a default purpose; the user's file is authoritative.
3. **Routing is host-controlled, but auditable.** The host decides *how* to score routing; the RFC defines only the *shape* of inputs (`routing_rules`, `capabilities`, `preferred_domains`) and the *shape* of outputs that get written back (`active_profile`, `profile_switch_log` entries with confidence + evidence references).
4. **No raw prompts in core.** Routing evidence is structured (rule matches, tag hits, capability identifiers). The raw user prompt that triggered the switch is **never** written into `usage_profile`. This is the moral equivalent of RFC-006 §6's no-PII invariant, applied to the user's own profile.
5. **User wins on gate conflict.** A `usage_profile` MAY define `recommended_verification_gates` per purpose (e.g. `health` → `confirm` on factual-claim-about-person). The user's top-level `verification_gates` (RFC-002) remain authoritative; the recommendation is a *default the user MAY accept by silence*, never a floor.
6. **Append-only switch log.** `profile_switch_log` is monotonically appended. Entries are never rewritten or deleted by an agent; they MAY be compacted by a future RFC-005-style compactor that preserves provenance.
7. **One active profile at a time.** `active_profile` is a single string identifier, not a set. Multi-skill blending is achievable via `routing_rules` deciding *which* single profile is active for the current turn, then switching cleanly on the next turn. This keeps audit logs tractable.
8. **GA does not block on this.** `v4.0.0` ships without `usage_profile`. If the empirical pressure is real (first-time users overwhelmed by bloat; routing inconsistency across providers), this RFC is promoted post-GA. See §13.

## 5. Shape (illustrative — non-normative)

```jsonc
  "klickd_version": "4.0",
  "profile_kind": "learner",
  "usage_profile": {
    "version": "1.0.0",
    "active_profile": "developer",
    "available_profiles": ["learner", "developer"],
    "capabilities": ["code_review", "debugging", "explain_concept"],
    "preferred_domains": ["python", "typescript", "distributed_systems"],
    "tool_permissions": {
      "allowed": ["web_search", "code_runner_sandbox"],
      "blocked": ["payment", "outbound_email"],
      "require_confirmation": ["web_search"]
    },
    "routing_rules": [
      {
        "id": "rule-legal-disclaimer",
        "match": { "intent_tags": ["legal", "contract", "compliance"] },
        "suggest_profile": "legal",

      },
      {
        "id": "rule-creative-media",
        "match": { "intent_tags": ["video", "reel", "tiktok", "thumbnail"] },

      "factual_claim_about_person": "confirm",

    },

      {
        "at": "2026-05-24T09:14:00Z",
        "from": "learner",
        "to": "developer",
        "trigger": "user_explicit",
        "evidence_ref": "rule-onboarding-default",
        "confidence": 1.0
      },
      {
        "at": "2026-05-24T11:02:33Z",
        "from": "developer",
        "to": "legal",
        "trigger": "rule_match",
        "evidence_ref": "rule-legal-disclaimer",
        "confidence": 0.82
      }
    ],
    "provenance": {
      "first_run_at": "2026-05-24T09:14:00Z",
      "first_run_purpose": "developer",
      "spec_revision": "SPEC.md@<git-sha>",
      "rfc_revision": "RFC-007@Draft"
    }
  }

```

### Field rules (intent — formal schema only if/when promoted past `Proposed`)

- `usage_profile.active_profile` MUST be a member of `usage_profile.available_profiles`.
- `usage_profile.available_profiles[]` MUST be drawn from the v1 enum (§7) or a documented `x_purpose_*` extension.
- `usage_profile.routing_rules[].suggest_profile` MUST be a member of `available_profiles`.
- `usage_profile.routing_rules[].min_confidence` is in `[0.0, 1.0]`. A host MUST NOT switch profile when the computed confidence is below this threshold.
- `usage_profile.profile_switch_log[]` is **append-only**: writers MUST NOT mutate or delete prior entries. Compaction is deferred to a future RFC (RFC-005-style).
- `usage_profile.profile_switch_log[].evidence_ref` MUST reference a structured identifier (rule id, capability tag, explicit user action). It MUST NOT contain free-form user text. (See §6.)
- `usage_profile.recommended_verification_gates` MUST use the same level vocabulary as RFC-002. On conflict with the user's top-level `verification_gates`, the **user wins**.

## 6. The no-raw-prompts invariant (normative intent)

The dual of RFC-006 §6 (no PII in `agent_core`) applies here in mirror form: **no raw user prompts in `usage_profile`**. Concretely:

- **Producer rule.** A writer that emits a `profile_switch_log` entry MUST set `evidence_ref` to a structured identifier (rule id, capability id, explicit user action like `user_explicit` / `user_command`). It MUST NOT embed the user's typed prompt, paraphrased intent, or any free-form natural-language excerpt. A producer that attempts to embed such material MUST surface `KLICKD_E_USAGE_PROFILE_RAW_PROMPT` and abort.
- **Reader rule.** A reader presented with a `profile_switch_log[].evidence_ref` containing apparent natural-language prose (heuristic: contains spaces, is longer than 128 characters, or fails a simple identifier regex) MUST surface `KLICKD_E_USAGE_PROFILE_EVIDENCE_SHAPE` and treat the entry as suspect (it MAY still load the file, but MUST flag the entry in any audit surface).
- **Validator rule.** A `.klickd validate` invocation MUST run the evidence-shape check independently of schema validity.

The intent is simple: a third party reading the user's `.klickd` (e.g. for a migration, an audit, a portability dump) sees *that* a switch happened, *what* rule fired, and *how confident* the host was — but does **not** see the user's private prompt. The user's prompts stay where they belong: in the session, not in the portable core.

This invariant is what makes the routing feature compatible with the broader `.klickd` privacy posture (SPEC §29c Privacy Guards, RFC-006 §6).

## 7. Starter purpose enum (illustrative — v1)

`usage_profile.active_profile` and `usage_profile.available_profiles[]` draw from a small starter enum in v1. Hosts MAY extend with `x_purpose_*` identifiers documented out-of-band.

| Purpose id | Intent | Typical sections used | `profile_kind` compatibility |
|---|---|---|---|
| `learner` | Default; education, growth, self-tracking. | `memory`, `growth`, `personality`, `onboarding_trigger`. | `learner` |
| `creator` | Media generation, scripts, reels, identity-of-voice. | `media_profile` (RFC-001), `verification_gates` (RFC-002), `claim_sources` (light). | `learner`, `team` |
| `developer` | Code review, debugging, system design. | `memory`, `tool_permissions`, no `media_profile`. | `learner`, `team` |
| `legal` | Contract review, compliance research, disclaimers. | `claim_sources` (strict), `verification_gates` (`confirm`/`block` on factual claims). | `learner`, `team` |
| `finance` | Personal finance, investing literacy, planning. | `claim_sources`, `verification_gates` (`warn`/`confirm`), no public-post defaults. | `learner` |
| `health` | Health literacy, symptom triage *to non-emergency information*. | `verification_gates` (`block` on diagnosis claims), `safety_policy` references. | `learner` |
| `research` | Literature review, claim grounding, citation discipline. | `claim_sources` (strict), `verification_artifacts` (RFC-002 §8b future). | `learner`, `team` |
| `agent_core` | The file *is* an agent operating context (RFC-006), not a user profile. **`usage_profile` SHOULD NOT be present in this case** — see §8. | n/a (RFC-006 applies) | `agent` |
| `mixed` | User explicitly opts into multi-purpose; host MUST list active routes. | union of selected purposes. | `learner`, `team` |

> **The enum is intentionally short.** The goal is not to enumerate every domain in the world; it is to keep the first-run choice *human*. Extensibility is via documented `x_purpose_*` extensions, not by inflating the enum.

## 8. Relation to `profile_kind` (SPEC §33.3) and RFC-006

`profile_kind` answers: *what role does this entire file play?* (`learner` / `agent` / `team` / `robot`).
`usage_profile.active_profile` answers: *within that role, which facet is engaged right now?*

The two interact as follows:

- A file with `profile_kind: "agent"` (RFC-006 `agent_core`) **SHOULD NOT** carry a `usage_profile` — an agent core has its own purpose by construction (the publisher's pedagogy/posture). If a producer needs per-tenant routing for an agent, that lives in the *deployer's* configuration, not in the published agent core. A reader presented with both `agent_core` and `usage_profile` MUST surface `KLICKD_E_USAGE_PROFILE_ON_AGENT_CORE` and refuse to load the file as either shape — same fail-closed posture as RFC-006 §6.
- A file with `profile_kind: "learner"` / `"team"` / `"robot"` MAY carry `usage_profile`. The compatibility table in §7 governs which `active_profile` values are admissible.
- A host MUST NOT silently coerce `profile_kind` based on `usage_profile.active_profile`. The two are set independently; mismatches surface as errors, not as auto-corrections.

## 9. First-run UX contract (illustrative — non-normative)

When a generator creates a brand-new `.klickd` for a user with no prior file, the SUGGESTED flow is:

1. **Ask the purpose, in plain language.** Example prompt:
   > "What do you want this `.klickd` to help you with first? You can change this later and add more. Choices: learner · creator · developer · legal · finance · health · research."
2. **Write the minimum.** Set `usage_profile.active_profile`, `available_profiles = [chosen]`, and seed only the sections that purpose needs (per the §7 table). Do not preallocate `media_profile`, `claim_sources`, etc., for purposes that do not need them.
3. **Record provenance.** Set `usage_profile.provenance.first_run_at` and `first_run_purpose`. These are immutable.
4. **Offer extension, do not impose it.** After first-run, the user MAY add a second purpose; this appends to `available_profiles` and seeds the new sections **only on demand**.
5. **No coerced consent.** Sections the user did not opt into are absent, not present-and-empty. Absent means "the user has not consented to this surface yet"; present-and-empty means "the user opted in and left it blank". Conflating the two breaks the GDPR posture documented in RFC-006 §10 and SPEC §29c.

This RFC does **not** mandate the prompt wording; it mandates the *shape* of the resulting `usage_profile` and the *invariant* that the base file stays minimal.

## 10. In-session skill routing contract (illustrative — non-normative)

When a `.klickd`-aware host serves a turn, it MAY consult `usage_profile` to decide which facet to engage. The SUGGESTED flow is:

1. **Compute candidate matches.** For each `routing_rules[]` entry, the host evaluates whether the current user intent (structured tags the host derives — *not* the raw prompt) matches `match.intent_tags`.
2. **Score confidence.** The host's internal scoring produces a confidence in `[0.0, 1.0]`. The scoring algorithm is host-internal and out of `.klickd` scope.
3. **Apply threshold.** If `confidence >= rule.min_confidence`, the rule is a candidate. If multiple rules match, the host SHOULD pick the one with the highest confidence; ties SHOULD be broken deterministically (lexicographic on `rule.id`).
4. **Switch — or stay.** If the winning candidate's `suggest_profile` ≠ current `active_profile`, the host MAY switch. If switching, the host MUST:
   - update `usage_profile.active_profile`;
   - append a `profile_switch_log[]` entry with `at`, `from`, `to`, `trigger: "rule_match"`, `evidence_ref: <rule.id>`, `confidence`;
   - surface the switch to the user with at least the rule id and the confidence (no opaque switches).
5. **Respect user override.** A `trigger: "user_explicit"` entry (e.g. `/profile legal`) MUST be honored unconditionally and SHOULD set confidence `1.0`.
6. **Never lower a gate.** The new profile's `recommended_verification_gates` MAY *raise* the effective gates above the user's top-level `verification_gates`, but a recommendation that would *lower* a user gate is dropped silently and logged as `KLICKD_W_USAGE_PROFILE_GATE_LOWERED` (warning, not error — the user's gate stays in force).
7. **No raw prompt leak.** The host MUST NOT write the user's prompt, paraphrase, or transcript snippet into the log. (See §6.)

This contract is host-controlled — the scoring, tag extraction, and prompt wording are all out of scope. The RFC only fixes the *inputs the host reads* and the *outputs the host writes back*.

## 11. Relation to RFC-002 and RFC-006

| Relation | Rule |
|---|---|
| `usage_profile.recommended_verification_gates` vs user `verification_gates` (RFC-002) | User wins on conflict. Recommendations are defaults the user MAY accept; never floors. |
| `usage_profile` vs `agent_core` (RFC-006) | Mutually exclusive per file. A file with both MUST be rejected (`KLICKD_E_USAGE_PROFILE_ON_AGENT_CORE`). |
| `usage_profile.tool_permissions` vs `agent_core.tool_policy` (RFC-006) | When a host loads both an agent core *and* a user `.klickd` with `usage_profile`, the **intersection** is the effective allow-list. Either party can *block* a tool; neither can *unblock* what the other blocked. |
| `usage_profile.profile_switch_log` vs RFC-005 (planned, claim-memory growth) | The same compaction discipline (provenance-preserving, never collapsed to prose) applies once RFC-005 exists. Until then, no compaction. |

## 12. Validation checklist (informative, for future v4+ conformance pack)

A conformance pack that targets `usage_profile` SHOULD include at least:

- [ ] **U-001** — Reject a file with `agent_core` and `usage_profile` (`KLICKD_E_USAGE_PROFILE_ON_AGENT_CORE`).
- [ ] **U-002** — Reject `usage_profile.active_profile` that is not a member of `available_profiles`.
- [ ] **U-003** — Reject `usage_profile.routing_rules[].suggest_profile` that is not a member of `available_profiles`.
- [ ] **U-004** — Reject a `profile_switch_log[]` entry whose `evidence_ref` looks like free-form natural language (`KLICKD_E_USAGE_PROFILE_EVIDENCE_SHAPE`).
- [ ] **U-005** — Reject a writer call that attempts to embed user prompt text in `profile_switch_log` (`KLICKD_E_USAGE_PROFILE_RAW_PROMPT`).
- [ ] **U-006** — Reject `min_confidence` outside `[0.0, 1.0]`.
- [ ] **U-007** — On gate conflict, the user's top-level `verification_gates` MUST win; a recommendation that lowers a user gate is dropped and a `KLICKD_W_USAGE_PROFILE_GATE_LOWERED` warning is emitted.
- [ ] **U-008** — A v3.x reader presented with `usage_profile` MUST preserve it verbatim on round-trip (no partial parse).
- [ ] **U-009** — A v4 reader without `usage_profile` support MUST preserve the field verbatim.
- [ ] **U-010** — A migrator (RFC-004) MUST NOT write `usage_profile` without explicit user confirmation; a default-on migration is non-conformant.
- [ ] **U-011** — `profile_switch_log` is append-only; rewriting a prior entry is non-conformant.

## 13. Error & warning codes (additions)

| Code | HTTP | Meaning |
|---|---|---|
| `KLICKD_E_USAGE_PROFILE_ON_AGENT_CORE` | 422 | A file carries both `agent_core` (RFC-006) and `usage_profile`. |

| `KLICKD_E_USAGE_PROFILE_EVIDENCE_SHAPE` | 422 | A `profile_switch_log[].evidence_ref` is not a structured identifier. |

| `KLICKD_W_USAGE_PROFILE_GATE_LOWERED` | 200 | A recommended gate that would lower a user gate was dropped; warning surfaced. |

## 14. Priority slicing (P0 / P1 / P2)

This RFC is **post-V4 GA** by construction (per §3 and §4 #8). Within its own scope, the suggested slicing is:

- **P0 (must-have if/when RFC is promoted past `Draft`):**
  - `usage_profile.active_profile` + `available_profiles[]` + starter enum (§7).
  - Progressive-disclosure principle (§9 #2).
  - No-raw-prompts invariant (§6).
  - Mutual exclusivity with `agent_core` (§8).
- **P1 (adoption-blocking once P0 is shipped):**
  - `routing_rules[]` + `profile_switch_log[]` (append-only, structured evidence).
  - `recommended_verification_gates` with user-wins resolution.
  - First-run UX contract (§9) referenced by [`docs/ux/V4-UX-SPEC.md`](../ux/V4-UX-SPEC.md).
- **P2 (extension / future-work):**
  - `x_purpose_*` extension discipline + a non-normative registry note.
  - Compaction of `profile_switch_log` (depends on RFC-005).
  - Cross-provider routing-consistency measurement (extension of RFC-003 Context Cost Benchmark).
  - Team-level shared routing rules (depends on RFC-001 / RFC-004 group-key work).

**None of P0/P1/P2 blocks `v4.0.0` GA.** All slicing is conditional on this RFC being promoted past `Draft` after `v4.0.0` ships.

## 15. Future work (post-v1)

- **Per-purpose schema fragments.** Once the enum stabilises, each purpose MAY get a fragment schema (e.g. `usage_profile-developer.schema.json`) declaring its required minimal sections. v1 keeps this implicit.
- **Routing telemetry, opt-in.** A user MAY opt into a local-only routing-quality log (which rule fired, was the switch reverted) for self-tuning. Strictly local; no upload. Likely RFC-008.
- **Multi-active blending.** Currently §4 #7 mandates one active profile at a time. A future RFC MAY introduce a weighted blend if real fixtures show single-active is too brittle.
- **First-party showcases.** Klickd MAY publish illustrative user `.klickd` fixtures per purpose (developer, learner, creator) in `examples/v4/usage_profile/` — same showcase pattern as RFC-006 §13 with `core.Kai.klickd`. Not part of v1 of this RFC.

## 16. Open questions

1. **Single `active_profile` vs weighted blend.** Today: single, switch cleanly. Open: do real users *want* the host to silently blend (e.g. legal + developer for "review this contract template's API surface")? Current draft: no — too hard to audit. Revisit if fixtures show single-active is impractical.
2. **`mixed` purpose semantics.** §7 lists `mixed` as a starter purpose. Is `mixed` a real value, or just shorthand for "available_profiles has > 1 entry"? Current draft: `mixed` is a *first-run convenience* (user picks "I don't know yet"); thereafter the file simply lists multiple `available_profiles`. The `active_profile` is still always a single concrete value.
3. **Where do recommended gates live?** Currently inside `usage_profile.recommended_verification_gates`. Alternative: a global `gate_recommendations` keyed by purpose. Current draft: keep it inside `usage_profile` to preserve one-section-per-concern; revisit if fixtures show cross-cutting recommendations.
4. **Migration default.** §4 #4 forbids auto-write of `usage_profile` on migration. Should the migrator at least *propose* `learner` as the default for a v3.x file (which is implicitly `learner` per SPEC §33.3)? Current draft: yes — propose, never auto-apply. The migration report (RFC-004) lists the proposal; the user confirms.
5. **Confidence semantics.** `min_confidence` is host-defined. Is a normative confidence calibration meaningful here? Current draft: no — calibration is a host concern; `.klickd` only carries the threshold the user (or default) set, and the recorded confidence at switch time, for audit.
6. **Privacy of `intent_tags`.** Tags themselves could leak intent ("intent_tags": ["divorce", "custody"]). Is the tag vocabulary normative? Current draft: no — tag vocabulary is host-defined; users can review their own routing rules and remove tags they consider sensitive. The no-raw-prompts invariant (§6) is the harder boundary.
